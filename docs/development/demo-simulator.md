# Demo Simulator — Architecture

The demo simulator is a procedural traffic generator inside the
`synapse-waf` binary that drives the real WAF detection engine and
shared managers from synthetic attacker traffic. When the binary is
launched with `--demo`, the simulator populates the same state
(`EntityManager`, `CampaignManager`, `BlockLog`, `HorizonManager`) that
production would populate from real network traffic, so every admin
endpoint Horizon polls — and the WebSocket push path real sensors use
— sees live-looking data without anyone actually sending HTTP requests
to the proxy.

This document covers the architecture. For recovery recipes when
something goes wrong in the demo pipeline, see
[demo-troubleshooting.md](./demo-troubleshooting.md).

## Goals

1. **High-fidelity demo mode**: exercise the real detection engine and
   real correlation detectors rather than returning canned JSON. A
   dashboard screenshot from the demo should be indistinguishable from
   one from production traffic apart from the source IPs.
2. **Foundation for a rule studio**: future phases will fork per-session
   engine instances from this same code path so analysts can edit rules
   and replay corpus slices against their drafts. Keeping the simulator
   inside the same binary as the engine makes the studio work small.
3. **Zero Horizon-side changes**: the simulator pretends to be a
   normal Synapse sensor, so Horizon's existing polling adapter and
   WebSocket ingestion work unchanged.

## Pipeline

```
┌────────────────────────────────────────────────────────────────┐
│                      synapse-waf binary                        │
│                                                                 │
│  ┌──────────────┐   ┌──────────────────┐   ┌───────────────┐   │
│  │  Archetype   │──▶│ DetectionEngine  │──▶│ EntityManager │   │
│  │  (CredStuff) │   │ analyze_with_    │   │ CampaignMgr   │   │
│  │  (VulnScan)  │   │ signals()        │   │ BlockLog      │   │
│  └──────────────┘   └──────────────────┘   │ WafStats      │   │
│         ▲                                  └───────┬───────┘   │
│         │                                          │           │
│  ┌──────┴───────┐                                  │           │
│  │ SimulatorLoop│                                  │           │
│  │ tick → gen → │                                  │           │
│  │ process_req  │                                  │           │
│  └──────┬───────┘                                  │           │
│         │  report_signal()                         │           │
│         ▼                                          │           │
│  ┌──────────────┐                                  │           │
│  │HorizonManager│  (WebSocket client)              │           │
│  └──────┬───────┘                                  │           │
└─────────┼──────────────────────────────────────────┼───────────┘
          │                                          │
          │ ws://localhost:3100/ws/sensors           │ HTTP
          ▼                                          ▼
     Horizon API                             Horizon synapse-direct
     (sensor-gateway.ts)                     (polls :6191 admin API)
          │                                          │
          ▼                                          ▼
       Prisma                               Live aggregate panels
       PostgreSQL
          │
          ▼
   Dashboard panels
```

Two distinct paths reach Horizon:

- **Push**: `SimulatorLoop.process_request` → `HorizonManager.report_signal` →
  batch → WebSocket send → `sensor-gateway.ts` → Prisma `signals` table →
  actor/campaign/threat dashboards.
- **Poll**: Horizon's `synapse-direct.ts` adapter polls the WAF admin
  API at `:6191` (`/health`, `/stats`, `/waf/stats`, `/metrics`) every
  few seconds → aggregate panels (live block rate, analyzed counters,
  DLP totals).

The push path is real-time per signal (with a
`signal_batch_delay_ms` flush cadence, default 1000ms). The poll path
is only aggregate numbers; it can't carry per-entity detail — that's
why waking up the HorizonManager push path was necessary to light up
the per-actor dashboards.

## Module layout

File: `apps/synapse-pingora/src/simulator.rs`

- **`SyntheticRequest`** — a single HTTP request generated by an
  archetype. Holds method, URI, headers, optional body, source IP,
  and an optional pre-computed JA4 string.
- **`Archetype` trait** — `fn next_request(&mut self) -> SyntheticRequest`.
  Archetypes own their own state machines (IP cursors, payload
  cursors) and produce one request per call.
- **`CredentialStuffer`** — 20 source IPs in `198.51.100.0/24` sharing
  one JA4 fingerprint (`t13d251200_a1b2c3d4e5f6_000000000000`, shaped
  like a Go `http.Client` default), cycling usernames against
  `POST /api/login`. Exercises the JA4 IP-cluster correlation detector.
- **`VulnScanner`** — one source IP (`203.0.113.99`), a sqlmap-shaped
  User-Agent, and 11 canonical attack payloads (SQLi, XSS, path
  traversal, command injection, recon). Trips production blocking
  rules (200200, 280001, etc.) and accumulates entity risk rapidly.
- **`SimulatorLoop`** — owns the archetype list, holds `Arc`s to all
  the shared managers, ticks on a `tokio::time::interval`, picks one
  archetype round-robin per tick, asks it for
  `requests_per_tick` requests, and runs each through
  `process_request`. Spawned once at startup via `loop.start()`.

## process_request: the seam

`process_request` is the single seam between synthetic traffic and the
rest of the proxy. It is deliberately small so drift from production
is easy to audit.

```rust
fn process_request(&self, req: SyntheticRequest) {
    // 1. Build ClientFingerprint from the archetype's JA4 + computed JA4H
    // 2. Call DetectionEngine::analyze_with_signals → DetectionResult
    // 3. Record stats: waf_stats.record(blocked, detection_time_us)
    // 4. Mirror state updates that request_filter would have done:
    //    - entity_manager.touch_entity_with_fingerprint
    //    - campaign_manager.register_fingerprints
    //    - (if blocked) block_log.record + apply_external_risk
    // 5. Emit ThreatSignal via horizon_manager.report_signal (if wired)
}
```

### Why mirror state updates instead of driving the full filter chain?

`request_filter` is bound to Pingora's `Session` and its socket
lifecycle. Faking a `Session` works (see
`tests/filter_chain_integration.rs`) but takes ~50 lines of plumbing
per call. The simulator doesn't need any of the network-layer work —
it just needs the engine verdict and the downstream state updates.
Mirroring them explicitly is ~10 lines and keeps the hot path cheap.

**The risk**: if production's `request_filter` grows a new state-update
call and the simulator is not updated, demo dashboards will silently
show less data than production. The mitigation is a periodic audit of
the two code paths. A longer-term cleanup is to extract
`process_request_state(request, verdict, &mut deps)` as a shared helper
both paths call — not done yet, scope creep for the MVP.

## HorizonManager wake-up

Before the demo work, `main.rs` constructed `SynapseProxy` with
`horizon_manager: None, // HorizonManager not yet initialized in main`
at both `ProxyDependencies` call sites. The `horizon::client`
library — 1600+ lines of WebSocket client with full auth, batching,
reconnect, circuit breaker — was compiled into the binary but never
instantiated in production either. The `if let Some(ref horizon) =
self.horizon_manager` branch in `request_filter` (around line 2022)
was unreachable.

The simulator work surfaced this because the dashboards that show
per-entity data are populated from the push path, not the poll path.
Without HorizonManager wired, nothing reached the WebSocket sensor
gateway, and those panels stayed empty.

The fix lives in `main.rs` near the admin-callback registration block:

```rust
let shared_horizon_manager: Option<Arc<HorizonManager>> =
    if config.horizon.enabled {
        // Spawn a dedicated thread with a multi-threaded tokio runtime,
        // construct HorizonManager::new().await, call .start().await,
        // send the Arc back via sync_channel, then park the runtime
        // forever on std::future::pending() so the WebSocket
        // background tasks stay alive.
    } else { None };
```

Both `ProxyDependencies` call sites now clone this `Option<Arc<…>>`
into their `horizon_manager` field. The `SimulatorLoop` gets the
same `Arc` and mirrors the `request_filter` ThreatSignal emit in its
own `process_request`.

### Sensor auth caveats

Horizon's sensor gateway authenticates sensors by looking up an
existing `Sensor` row by `(tenantId, name)` and comparing the
`SensorApiKey` hash plus an optional `fingerprint` column. The dev
seed (`apps/signal-horizon/api/prisma/seed/seed-postgres.ts`)
creates a sensor named **`Synapse WAF`** with id **`synapse-waf-1`**,
api key **`sk-sensor-bridge-dev`**, and a computed fingerprint.

`config.horizon.yaml` in this repo is pinned to those exact values.
If you see `Auth failed: New sensors must use a registration token`,
the `sensor_name` in the config does not match the seeded row. If you
see `Auth failed: Sensor identity verification failed`, the gateway
found the row but the `fingerprint` column doesn't match what the
client sends — a known hole: `synapse-pingora`'s horizon client does
not compute the same formula the seed uses. Recovery is to null the
fingerprint column (see the troubleshooting doc) so first-connect
repopulates it.

## Tuning knobs

The simulator's defaults:

| Knob | Default | Effect |
|---|---|---|
| `tick_interval` | 1000 ms | How often the loop fires |
| `requests_per_tick` | 2 | Requests per archetype per tick |
| Archetype count | 2 | `CredentialStuffer` + `VulnScanner` |

Effective RPS = `(1000 / tick_interval_ms) × requests_per_tick` per
archetype, rotated round-robin. The defaults above produce ~4 RPS
which is sustainable indefinitely against a dev Horizon API.

**Do not increase beyond 10 RPS on a dev box.** Each simulator
request produces a ThreatSignal, which Horizon's aggregator processes
with an INSERT + SELECT + idempotency round-trip per signal. At
20 RPS the Node event loop saturates after a few minutes and the
dashboard hangs at "Connecting..." — not because the simulator
crashed but because `/ws/dashboard` upgrade requests can't get past
the ingest queue. Detailed recovery in the troubleshooting doc.

A Phase 2 candidate is virtual-clock injection so demos can
fast-forward cleanly. That is not implemented yet; the loop uses
`tokio::time::interval` against wall-clock time.

## Distinguishing simulator data from real data

The simulator is built on reserved IP ranges and explicit tagging so
analysts can always tell demo from real:

- **RFC 5737 documentation blocks**: source IPs are drawn from
  `198.51.100.0/24` (TEST-NET-2) and `203.0.113.0/24` (TEST-NET-3).
  These ranges are guaranteed never to appear in real internet
  traffic. If you see them anywhere in a dashboard, the row is
  simulator-origin with 100% certainty.
- **Metadata tag**: every ThreatSignal emitted by the simulator
  carries `metadata.source = "simulator"` in its JSON body. Query
  `SELECT metadata->>'source' FROM signals` to filter.
- **Block reason**: `BlockEvent.reason` falls through to
  `"simulator"` when the engine doesn't produce a specific reason.
- **User agents**: `Go-http-client/1.1` (credential stuffer) and
  `sqlmap/1.7.2#stable (https://sqlmap.org)` (vuln scanner) are
  obvious tells in any panel that surfaces UA.

## Known limitations

1. **No per-session isolation**: the simulator drives the global
   shared engine, so a rule studio that lets visitors edit rules needs
   either per-session engine forks or an in-browser WASM build. Neither
   exists yet.
2. **Single tenant**: Horizon's dev seed creates three tenants (acme,
   globex, initech) but the simulator only produces data for the acme
   tenant (via the seeded `synapse-waf-1` sensor). Cross-tenant demos
   would need a second simulated sensor.
3. **Campaign ID namespace schism**: Horizon's campaigns list mixes
   two data sources — seeded database campaigns (Prisma cuids like
   `cmny0w8br09qk680xpo9b7au6`) and live synapse-waf-local campaigns
   (ids like `camp-19d89d644a0`). The detail page fetches from
   synapse-waf's admin API, so seeded cuids 404 on click. See
   troubleshooting for the workaround.
4. **No archetype diversity yet**: only CredentialStuffer and
   VulnScanner ship. Follow-ups: noisy benign user (real browser-shaped
   traffic that does NOT block), DLP-leaking insider (POST with SSN
   and credit card numbers), bot scraper, impossible-travel session.
5. **HorizonManager fingerprint mismatch**: the seeded
   `sensors.fingerprint` column does not match what synapse-waf's
   horizon client sends, so first-connect currently requires manually
   nulling the column. Production fix is to teach the client to
   compute the same formula the seed uses, or to have the seed leave
   the column null and rely on first-connect to populate it.

### Release build vs. debug build

`just demo` uses the **release** binary
(`target/release/synapse-waf`), not the debug binary. This is
deliberate and load-bearing for any demo session longer than about
half an hour.

- **Debug build**: Rust's `Drop` paths are not optimized away in debug
  mode, and the simulator's long-running state (EntityManager history,
  FingerprintIndex, BlockLog, Ja4RotationDetector observations,
  SCHEMA_LEARNER templates) accumulates toward its bounded caps.
  Resident memory climbs over 30-60 minutes until macOS's OOM killer
  picks the process and SIGKILLs it. Symptom in the tmux window:
  `zsh: killed ... ✘ KILL took 37m 26s` followed by return to a shell
  prompt. No panic, no stack trace — the process simply disappears.
- **Release build**: `Drop` is fully elided, inlining shrinks the
  per-entry overhead on the bounded caches, and the process steady-
  states well below the OOM threshold. A release binary has been
  observed running indefinitely under sustained simulator load.

If you're iterating on the `simulator.rs` code itself, use
`just dev-synapse` which compiles and runs the debug binary (faster
edit-compile-run loop). Switch back to `just demo` when you're done
and want to leave the demo running.

### Teardown

All three services run as tmux windows inside the `edge-protection`
session by default (`TMUX_SESSION` env var overrides the session
name). The standard dev-session controls apply:

- `just dev-stop` — stop all three windows
- `just dev-stop-one synapse-pingora` — stop only the WAF
- `just dev-restart` — full restart
- `just dev-reset` — kill and recreate the tmux session itself
  (useful if a window gets wedged)
